## Section 1 — System Design & Architecture Decisions

**Q: Why did you build a custom integrity scoring system instead of using an existing proctoring tool like Proctorio or Honorlock?**
A: Existing tools rely on webcam/microphone capture, which creates GDPR Article 9 biometric processing obligations that require explicit consent and Data Protection Impact Assessments specific to that data category. Our institution's legal team had not cleared that basis. AEGIS uses behavioural metadata only — keystroke intervals, tab events, paste events — which falls under legitimate interest for academic integrity under Article 6(1)(f). Beyond GDPR, commercial tools are black boxes: a professor cannot explain to a student why they were flagged. AEGIS produces a per-signal breakdown (tab_switch_score: 0.45, paste_score: 0.80) that is directly explainable and auditable.

**Q: Walk me through a request from a student pressing a key in their exam browser to a risk score appearing on the professor's console.**
A: The browser SDK captures the inter-keystroke interval in milliseconds and batches it into a key_interval event payload. The WebSocket send call is fire-and-forget from the student's perspective — it never blocks answer input. On the server, the telemetry WebSocket handler in `app/routers/telemetry.py` receives the raw JSON, calls `live_monitor.record_event()` which updates an in-memory StudentAggregate with a running mean IKI, and calls `telemetry_service.store_event()` which persists the row to `telemetry_events` in PostgreSQL. The professor console WebSocket loop runs every five seconds, calls `live_monitor.snapshot()` which is a pure memory read, and pushes the result as JSON over the professor WebSocket. No database query occurs on every professor tick — only on the initial roster seed.

**Q: What happens if the scoring service is unavailable when an exam closes?**
A: Exam closure (`POST /exams/{id}/close`) commits the state change to PostgreSQL first, unconditionally. It then calls `background_tasks.add_task(dispatch_score_job, exam.id)`, which runs after the HTTP response is sent. Inside `dispatch_score_job`, local scoring is attempted first via `compute_and_save_scores`. If that fails, the error is logged but does not propagate to the HTTP response. The Service Bus dispatch is wrapped in its own try/except and also fails silently. The exam is closed regardless. A professor can manually trigger rescoring by closing again or via a future admin endpoint. Telemetry events remain in the database and scoring can be rerun at any time against them.

**Q: How does the system handle a professor and student on opposite sides of the globe?**
A: The current deployment is single-region on Azure Container Apps. WebSocket latency from a geographically distant student would be higher, but this does not affect correctness: telemetry events carry both `client_ts` (browser timestamp) and `server_ts` (server receive time). The scorer uses `occurred_at` which is the server timestamp, so clock skew between student and server does not corrupt the scoring. For genuine multi-region support, the architecture would need regional WebSocket gateways relaying events to a central scorer via Service Bus, which is consistent with the existing decoupled design but not implemented in the current scope.

**Q: Why is the live monitor in-process memory rather than Redis?**
A: The live monitor is explicitly a best-effort, low-latency view for the professor. It does not need to survive a process restart — the authoritative data is in PostgreSQL. Putting it in Redis would add a network hop on every telemetry event, which at 200 concurrent students sending keystroke events every few seconds creates significant Redis write volume. More importantly, the live monitor only needs to be consistent for the single professor WebSocket connection. In a single-worker deployment (which is appropriate for exam-scale concurrency), in-process memory is correct. If the system scaled to multiple workers, the professor could be routed to the same worker as their students via sticky sessions, or the monitor could be moved to Redis at that point. The design makes that migration straightforward.

**Q: What is the data flow for GDPR consent and how is it enforced?**
A: When a student navigates to their exam, `GET /exams/{id}/session` creates a StudentSession row with `consent_at = NULL`. The frontend shows a consent screen. On acceptance, `POST /exams/{id}/consent` sets `consent_at` to the current UTC timestamp. `GET /exams/{id}/questions` checks for a StudentSession with non-null `consent_at` and returns 403 if absent. Telemetry WebSocket connection proceeds independently of consent — the student might connect before consenting — but telemetry events are stored regardless. The consent gate is on question access, not on telemetry. This means we do not inadvertently collect keystroke data before consent, because no questions are displayed before the gate is passed.

**Q: How would you scale AEGIS to 10,000 concurrent students?**
A: The current bottleneck is the single PostgreSQL instance for telemetry writes. At 10,000 students sending events every 2-3 seconds, that is 3,000-5,000 inserts per second, which exceeds single-instance PostgreSQL throughput. The fix is to decouple telemetry ingestion from persistence: the WebSocket handler would write to Service Bus only, not directly to PostgreSQL. A separate ingestion worker would batch-insert events to PostgreSQL using COPY or bulk insert. The live monitor would remain in-memory per worker, with Redis pub/sub broadcasting events to the professor's worker. The scorer would be unchanged since it runs post-exam. Azure Container Apps with KEDA scaling on Service Bus queue depth would handle the compute layer automatically.

**Q: Why did you choose a state machine (draft → open → closed) for exam sessions rather than timestamp-based status?**
A: Timestamp-based status (e.g., "exam is open if now() is between starts_at and ends_at") creates ambiguity at boundaries and makes manual intervention difficult. A professor may need to extend an exam, pause it, or reopen it for a student who had a technical issue. With explicit state, the state machine enforces valid transitions (`_TRANSITIONS` dict in `exam.py`) and rejects invalid ones with a 409. The `opened_at` and `closed_at` columns record when transitions occurred for audit purposes, but status is authoritative. This also means the system does not silently change state based on the clock, which would create race conditions between the state check and the database write.

**Q: What happens to in-flight answers if the server crashes mid-exam?**
A: PostgreSQL provides the durability guarantee. `POST /exams/{id}/answers` calls `await db.commit()` before returning 200. If the server crashes after the commit, the answers are in PostgreSQL and are not lost. If the server crashes before the commit, the transaction is rolled back by PostgreSQL automatically. The student's browser does not receive a 200, so the frontend can retry. The answer submission endpoint is idempotent: submitting the same answer twice upserts, it does not duplicate. The student's in-browser state (React component state) holds the draft answer until a successful 200 is received.

**Q: How does the system prevent a student from accessing another student's answers or telemetry?**
A: Every protected endpoint extracts the caller's identity from the JWT via `require_role()` dependency, which decodes the token and returns the `sub` claim as `student_id`. Answer retrieval (`GET /exams/{id}/answers`) filters by `WHERE student_id = <calling student's id>`. There is no endpoint that returns all students' answers — that is only accessible via `GET /exams/{id}/grade` which requires `require_role("professor")`. Telemetry events are stored with `student_id` but there is no retrieval endpoint for a student to query their own telemetry — it is write-only from the student's perspective. The professor endpoints verify ownership (`_assert_owner`) so a professor cannot access another professor's exams.

## Section 2 — Signal Scoring

**Q: Why are there six signals and not more or fewer?**
A: Six signals cover the primary behavioural vectors identified in the academic literature on browser-based proctoring. Tab switching and focus loss detect navigation away from the exam. Paste events detect content insertion from external sources. Keystroke interval deviation detects rhythm discontinuities consistent with dictation or copy-paste followed by typing. Answer timing detects implausibly fast responses. Window resize detects possible second-monitor or screen-share indicators. Adding more signals risks double-counting (e.g., copy sequence and paste are correlated) and reduces the explainability of individual scores. Fewer signals would miss meaningful patterns. The six were validated against the signal taxonomy in Romero et al. 2021.

**Q: Why does tab_switch carry a weight of 0.30, the highest of all signals?**
A: Tab switching is the most unambiguous signal: a student leaving the exam tab during an exam has no legitimate explanation in a closed-book context. Paste and keystroke deviation can have innocent causes (auto-correct paste, nervous typing). Tab switch cannot. The 0.30 weight means a student who tab-switches repeatedly will have a high integrity score regardless of their other behaviour, which reflects the severity of the signal. In lenient preset mode, this weight is reduced to accommodate open-book exams where referencing materials in other tabs may be permitted.

**Q: How does the IKI (inter-keystroke interval) score work and what is its known weakness?**
A: The IKI score computes the mean interval across all `key_interval` events for a student. The formula is `iki_score = clamp((400 - mean_iki_ms) / 400)`. A mean interval below 50ms produces a score near 1.0 (highly suspicious — no human types that fast consistently). A mean above 400ms produces a score near 0.0. The 400ms baseline was chosen from keystroke dynamics literature as the upper boundary of normal typing rhythm. The known weakness: this score is computed from the global mean, not from local bursts. A student who types normally for most of the exam and has one 500-keystroke burst at machine speed will have their burst diluted by the normal typing, producing a lower score than their actual behaviour warrants. The `key_burst` event type partially addresses this by flagging concentrated fast-typing windows, but the scorer does not yet integrate burst detection directly into the IKI component (tracked as AEGIS-55).

**Q: What is the StudentBaseline and how is it used?**
A: StudentBaseline stores per-student pre-exam typing statistics: `mean_keystroke_interval_ms` and `keystroke_stddev_ms`, computed from keystrokes in the first few minutes of the exam before any suspicious events are detected. The intent is to personalise the IKI scoring: a student who naturally types at 150ms average should not be flagged for their normal rhythm. Currently the baseline is stored but not yet integrated into the real-time scorer, which uses the global 400ms reference. Phase 3 will replace the global reference with `student_baseline.mean_keystroke_interval_ms + 2 * student_baseline.keystroke_stddev_ms` as the threshold, making the score relative to each student's own baseline.

**Q: How does the paste score distinguish between one accidental paste and systematic paste-based cheating?**
A: The paste component scorer receives all `paste` events and computes a score based on frequency and payload size. A single short paste (text_length < 20 characters) might be a password manager auto-fill or autocorrect — the scorer applies a low weight to short pastes. Multiple pastes with text_length > 100 characters are weighted more heavily. The exact implementation is in `app/services/scoring/components/paste.py`. The score is bounded 0-1 and feeds into the aggregate with a 0.25 weight. A professor reviewing a flagged student will see both the paste_score component and the raw telemetry events including each paste's text_length, so they can make a contextual judgement.

**Q: What is the resize signal actually detecting and can it be triggered accidentally?**
A: The `resize` event fires when the browser window dimensions change significantly during an exam. The rationale is that screen sharing software often resizes the browser to fit a shared screen. It can also indicate a student moving between monitors. It carries the lowest weight (0.05) because it is the noisiest signal — accidental resize by dragging the window edge, or a system notification pushing the window, will fire the event. The scorer therefore treats resize as a weak corroborating signal rather than a primary indicator. A high resize_score alone would not flag a student; it contributes meaningfully only when other signals are also elevated.

**Q: How do the three scoring presets (lenient, standard, strict) differ mechanically?**
A: The presets are stored in `scoringPresets.ts` on the frontend and sent as `scoring_preset` on exam creation. The backend stores the preset on `ExamSession.scoring_preset`. At scoring time, `compute_and_save_scores` reads the preset and selects a weight vector. Strict increases tab_switch weight to 0.40 and reduces answer_time weight. Lenient reduces tab_switch to 0.15 and increases the threshold for flagging from 0.70 to 0.80. Standard uses the baseline weights. The preset is fixed at exam creation and cannot be changed after the exam opens, preventing retroactive adjustment of results.

**Q: Why is the risk flag threshold 0.70 and not 0.80 or 0.60?**
A: 0.70 was chosen as a balance between false positive rate and recall. At 0.60, a student with moderate tab-switching and one paste event would be flagged — too sensitive for a first-generation system where the professor must review every flag manually. At 0.80, only students with extreme behaviour across multiple signals would be flagged — likely to miss borderline cases. 0.70 means a student needs substantial signal across at least two components to breach the threshold. This threshold is explicitly configurable via presets (0.70 for standard, 0.80 for lenient) and should be calibrated against real exam data in Phase 2.

**Q: What happens if a student has no telemetry events at all — no keystrokes, no tab events?**
A: A student with no telemetry produces all-zero component scores and an aggregate of 0.0. They will not be flagged. This can occur if the student's browser blocks the WebSocket connection (corporate firewall, browser extension), if they use an older browser that does not support the telemetry SDK, or if they answer the exam without typing (e.g., voice dictation). The scorer does not penalise absence of telemetry — absence of evidence is not evidence of cheating. However, the professor console will show the student as inactive, which is a visible signal for manual review. A future enhancement would flag students with zero telemetry for a different reason: they may have circumvented monitoring.

**Q: How does the system avoid double-penalising a student for a single behaviour that triggers multiple signals?**
A: There is partial correlation between signals — a paste event often precedes a keystroke burst, and a tab switch might precede a paste. The current scorer does not explicitly decorrelate signals; it computes each independently. The aggregate is a weighted sum, so if paste_score and iki_score are both elevated due to a single paste-and-type sequence, both contribute to the aggregate. This is a known limitation. The mitigation is that the weights sum to 1.0 and each component is bounded 0-1, so the aggregate cannot exceed 1.0 regardless of correlation. The Phase 3 ML approach would learn the correlation structure from data and reduce double-counting automatically.

## Section 3 — WebSocket Implementation

**Q: How would the system handle a WebSocket disconnect mid-exam?**
A: On the server side, `WebSocketDisconnect` is caught in the receive loop and the loop exits cleanly. The finally block in `exam_telemetry_ws` removes the student's entry from `_connections` and calls `_mark_disconnected`, which stamps `ws_disconnected_at` on the StudentSession row. On the client side, the browser SDK uses exponential backoff reconnection: 100ms, 200ms, 400ms, up to a maximum of 30 seconds. On reconnect, a new WebSocket is accepted by the server and the student's entry in `_connections` is replaced with the new socket. The student session remains open in the database; there is no penalty for reconnection. Buffered telemetry events from the client's circular buffer are replayed to the server on reconnect.

**Q: Why does the professor console use a 5-second polling loop over the WebSocket rather than server-push events?**
A: Server-push would require the server to track which students' data has changed since the last push and selectively broadcast. With 200 students, tracking per-student change state in the server adds complexity for minimal benefit: the professor console re-renders the full student list anyway because risk scores for all students are displayed simultaneously. A 5-second full snapshot is simpler to implement, simpler to debug, and acceptable for a monitoring use case where sub-second latency is not required. The tradeoff is that a risk flag that fires at t=0 may not appear on the professor console until t=5. The acceptance criteria specified "visible within 5 seconds of threshold breach," which this satisfies.

**Q: How does the heartbeat mechanism prevent zombie WebSocket connections?**
A: The `_heartbeat` coroutine sends `{"type":"ping"}` every 30 seconds over the student WebSocket. If the student's browser has gone away without a clean close (e.g., network cut, laptop lid closed), the TCP connection may remain half-open on the server. The `send_text` call will raise an exception when the TCP stack detects the broken connection, typically after the OS TCP keepalive timeout. The heartbeat ensures this happens within 30 seconds plus the TCP timeout rather than potentially never. The heartbeat task and the receive task run concurrently via `asyncio.wait(..., return_when=FIRST_COMPLETED)`. When either exits, the other is cancelled.

**Q: What prevents two professors from connecting to the same exam's professor WebSocket simultaneously?**
A: `_professor_connections` is a dict keyed by `exam_id_str`. When a new professor WebSocket connects, it replaces whatever was in the dict for that key. The previous connection is not explicitly closed — it continues to receive data until it disconnects naturally. In practice, a second professor connecting would shadow the first: the risk_alert push from the scorer would go to the second connection. This is a known limitation for the current single-professor-per-exam model. A production system would use a list or set of connections per exam and broadcast to all of them.

**Q: How does the server know which exam a WebSocket message belongs to?**
A: The exam ID is part of the URL path: `/ws/exam/{exam_id}`. FastAPI extracts it as a path parameter before the WebSocket upgrade handshake completes. The exam ID is validated as a UUID by the type annotation. Inside the receive loop, every `store_event` call and every `live_monitor.record_event` call is scoped to the `exam_id` captured at connection time. There is no per-message exam ID — the connection itself carries the context. This prevents a student from injecting telemetry for a different exam by sending a modified payload.

**Q: How does the system handle backpressure if a student's browser sends events faster than the server can process them?**
A: FastAPI's WebSocket receive loop is async: `await ws.receive_text()` yields control to the event loop while waiting for the next message. Processing each message (live monitor update + DB write) is also async. If the student sends faster than the server processes, messages queue in the TCP receive buffer. When the TCP buffer fills, the browser's WebSocket send call blocks. The browser SDK does not implement its own send queue — it drops events if the TCP buffer is full, which is acceptable for telemetry (it is best-effort). For answer submission, which uses HTTP POST not WebSocket, there is no such concern: the student waits for a 200 before considering the answer saved.

**Q: What is the `_registry_lock` and why is it needed?**
A: `_connections` is a dict of dicts that is read and written by multiple concurrent coroutines — the receive loop writes on connect/disconnect, and the scorer (via `get_professor_websocket`) reads on scoring. Without a lock, a coroutine reading `_connections` could see a partially-written state if another coroutine is modifying the same dict simultaneously. `asyncio.Lock` is used rather than `threading.Lock` because the code runs in a single-threaded async event loop. The lock is held only for the duration of the dict read or write, not for the duration of the WebSocket communication, so it does not create a bottleneck.

**Q: How are JWT tokens validated for WebSocket connections, given that WebSockets don't support HTTP headers after the upgrade?**
A: The token is passed as a query parameter: `?token=<jwt>`. This is the standard browser WebSocket workaround since the JavaScript WebSocket API does not allow setting custom headers. The `_decode_token` function in the telemetry router decodes and validates the token using `python-jose`. If the token is invalid or missing, the server accepts the WebSocket (necessary to send a close code) and immediately closes it with code 4401. The token is validated before any exam ID or enrollment checks, so an unauthenticated connection cannot probe which exam IDs exist.

**Q: What happens to the professor WebSocket if the scorer pushes a risk_alert but the professor has just disconnected?**
A: `_push_risk_alert` calls `get_professor_websocket`, which returns the WebSocket object stored in `_professor_connections`. If the professor just disconnected, the `finally` block in `professor_monitor_ws` has removed the entry, so `get_professor_websocket` returns `None`. `_push_risk_alert` logs a debug message and returns without error. If the professor reconnects before the push, they will receive the alert on reconnect via the next 5-second snapshot, which will include the flagged student's elevated score. The RiskFlag row is already in the database regardless of whether the push succeeds.

**Q: How would you test WebSocket endpoints in CI without a live browser?**
A: The existing test suite uses `httpx.AsyncClient` with `ASGITransport`, which supports WebSocket connections via `client.websocket_connect()`. This allows WebSocket tests to run against the ASGI app in-process without a live server. For the telemetry endpoint specifically, a test would connect as a student, send a JSON frame, disconnect, and assert that the `TelemetryEvent` row was created in the SQLite test database. The JWT for the test connection is minted inline using the same secret and algorithm as the app.

## Section 4 — FastAPI and Async Python

**Q: Why is SQLAlchemy's async session used instead of a synchronous session with threading?**
A: FastAPI runs on an async event loop (uvicorn with asyncio). A synchronous SQLAlchemy session calls block the thread. Since Python's asyncio runs on a single thread by default, a blocking database call would freeze the entire event loop — all WebSocket connections, all HTTP requests — until the database responds. SQLAlchemy 2.0's async session (`AsyncSession`) uses `asyncpg` under the hood, which is a native asyncio PostgreSQL driver. Every database call is awaited, yielding control to the event loop while the network round-trip completes. This allows 200 concurrent WebSocket connections to coexist without one student's database write blocking another's.

**Q: What is `await db.flush()` and when is it used instead of `await db.commit()`?**
A: `flush()` sends pending SQL statements to PostgreSQL within the current transaction, which causes PostgreSQL to assign generated values (like `gen_random_uuid()` defaults) and makes them available on the Python object. Critically, it does not commit the transaction — the changes are visible only within the current session, not to other connections. It is used in the seed script when an entity is created and its auto-generated UUID is needed immediately as a foreign key for the next entity. `commit()` makes changes durable and visible to all connections. `flush()` followed by `commit()` is the standard pattern when you need intermediate state within a single transaction.

**Q: How does Pydantic v2 differ from v1 and what breaking changes affected the project?**
A: The main breaking changes were: `orm_mode = True` became `model_config = ConfigDict(from_attributes=True)`. `.dict()` became `.model_dump()`. `.parse_obj()` became `.model_validate()`. `validator` became `field_validator` with different decorator syntax. The project uses `model_validate` throughout the routers (e.g., `ExamRead.model_validate(exam)`) and `model_dump(mode="json")` where JSON-serialisable output is needed. Pydantic v2 is significantly faster than v1 because the core validation logic is implemented in Rust (pydantic-core), which matters for endpoints that serialise large response bodies like grade reports.

**Q: What is the FastAPI dependency injection system and how does `require_role` use it?**
A: FastAPI's `Depends()` system allows declaring shared logic as functions that are called automatically before the endpoint function. `require_role("professor")` returns a dependency function that extracts the JWT from the `Authorization: Bearer` header, decodes it, checks the `role` claim, and returns the `user_id`. If any check fails, it raises an HTTPException which FastAPI catches and converts to an HTTP error response. The endpoint function receives `user_id: str = Depends(require_role("professor"))` and never needs to deal with JWT decoding itself. The dependency is cacheable within a single request and reusable across all endpoints that require the same role check.

**Q: How does `asyncio.wait` with `return_when=FIRST_COMPLETED` work in the WebSocket handler?**
A: `asyncio.wait` runs multiple coroutines concurrently and returns when a specified condition is met. With `FIRST_COMPLETED`, it returns as soon as any one of the tasks finishes, leaving the others in a pending state. In the student WebSocket handler, the receive loop and heartbeat run concurrently. If the student disconnects, the receive loop exits, `asyncio.wait` returns, and the finally block cancels the heartbeat task. If the heartbeat fails (broken connection), `asyncio.wait` returns, and the receive loop is cancelled. This pattern ensures both tasks are cleaned up regardless of which one exits first, preventing coroutine leaks.

**Q: Why does the scorer use `NullPool` for the database connection in Alembic migrations?**
A: `NullPool` disables connection pooling — every database operation opens a new connection and closes it when done. This is appropriate for migration scripts because migrations are run as a one-shot process, not as a long-running server. A connection pool in a one-shot process would hold connections open unnecessarily. For the application server, `AsyncSession` uses SQLAlchemy's default async pool (`AsyncAdaptedQueuePool`), which maintains a pool of reusable connections. The pool size should be tuned to match PostgreSQL's `max_connections` setting, which is 100 by default on Azure Database for PostgreSQL Flexible Server.

**Q: What does `expire_on_commit=False` mean in the session factory and why is it set?**
A: By default, SQLAlchemy expires all attributes on ORM objects after a commit. Accessing an attribute on an expired object triggers a new database query (lazy load). In async code, lazy loading is not supported — there is no implicit await, so the access would fail or raise a `MissingGreenlet` error. `expire_on_commit=False` disables expiry, meaning attribute access after a commit returns the cached value from the Python object without a database round-trip. The tradeoff is that the cached value may be stale if another process modified the row. For the test suite and most application code, this is acceptable because the object is either used immediately after commit or refreshed explicitly via `await db.refresh(obj)`.

**Q: How does the `get_db` dependency ensure the session is always closed, even if an exception occurs?**
A: `get_db` is an async generator function that uses `async with AsyncSessionLocal() as session: yield session`. The `async with` block on `AsyncSessionLocal()` calls `__aenter__` on the session context manager, which creates the session. When the generator is exhausted (after the endpoint function returns) or when an exception propagates through the `yield`, the `__aexit__` is called, which closes the session. FastAPI handles the generator lifecycle via its dependency injection cleanup mechanism. If the endpoint raises an HTTPException, FastAPI catches it, sends the error response, and then triggers dependency cleanup, so the session is closed even on error paths.

**Q: Why are there two timestamp columns on TelemetryEvent (`client_ts` and `server_ts`) instead of one?**
A: `client_ts` is the browser's reported timestamp — it reflects what the student's clock says at the moment the event occurred. `server_ts` is when the server received and stored the event. These can differ significantly: a student with a clock set an hour ahead, or a burst of events buffered during a reconnection that arrive all at once with their original timestamps. The scorer uses `server_ts` (via `occurred_at`) for ordering events chronologically in a reliable way. `client_ts` is preserved for forensic analysis — if a student claims they were not tab-switching at a certain time, `client_ts` provides their perspective, while `server_ts` provides the server's authoritative record.

**Q: What is a Pydantic discriminated union and could it be used for telemetry payloads?**
A: A discriminated union uses a shared field (the discriminator) to determine which of several models to validate against. For telemetry payloads, you could define `TabBlurPayload(duration_ms: int)`, `PastePayload(text_length: int, question_position: int)`, etc., and union them with `event_type` as the discriminator. This would provide type-safe payload validation per event type. AEGIS does not do this currently — payloads are stored as `JSON/JSONB` and validated loosely. The reason is pragmatic: adding a new event type requires a schema change in a discriminated union, whereas JSONB accepts any payload without migration. The tradeoff is that malformed payloads (e.g., a paste event missing `text_length`) are only discovered at scoring time when the scorer tries to read the field.

## Section 5 — Azure Infrastructure

**Q: Why Azure Container Apps and not Azure Kubernetes Service?**
A: AKS requires managing node pools, cluster upgrades, ingress controllers, certificate management, and pod security policies. Azure Container Apps abstracts all of this: a container image, environment variables, and a scaling rule are sufficient. The team has four people and a twelve-week timeline — AKS operational overhead is disproportionate. ACA supports WebSocket connections natively on its HTTP ingress. KEDA integration with Azure Service Bus allows the scorer worker to scale from zero replicas between exams to N replicas when queue depth increases, which optimises the $500 Azure for Students credit: the scorer costs nothing when no exam is running.

**Q: Why Azure Service Bus and not Azure Event Hubs?**
A: Event Hubs is a high-throughput streaming service designed for millions of events per second with replay capability. It uses a partition-based consumer model where consumers must track their own offset. Service Bus is a message queue with dead-lettering, message locking, and at-least-once delivery guarantees. For AEGIS, the scorer processes one batch per exam session — it does not need to replay historical events or process millions of events per second. Service Bus's dead-letter queue provides automatic handling of failed messages with configurable retry counts, which would need to be implemented manually on top of Event Hubs. Service Bus is also simpler to integrate with KEDA for ACA autoscaling.

**Q: How does KEDA autoscaling work with the scorer worker?**
A: KEDA (Kubernetes Event-Driven Autoscaling) monitors an Azure Service Bus queue and scales the scorer worker deployment based on queue depth. When an exam closes and a score job message is enqueued, KEDA detects the queue depth increasing from 0 to 1 and scales the scorer worker from 0 to 1 replica. The worker processes the message, the queue drains to 0, and KEDA scales the worker back to 0 after a cooldown period. This means the scorer worker has zero cost when no exams are being scored. The scale-up time is typically 30-60 seconds, which is acceptable since scoring runs after the exam is closed and does not need to complete instantly.

**Q: What is the difference between Azure Container Apps environments and revisions?**
A: An environment is the network boundary — all containers within an environment share the same virtual network and can communicate with each other via internal DNS. A revision is a snapshot of a container app's configuration (image tag, environment variables, scaling rules). When you deploy a new version of the app, ACA creates a new revision. Traffic can be split between revisions (e.g., 90% to stable, 10% to canary). For AEGIS, the telemetry gateway, exam API, and scorer worker are separate container apps within the same environment. This allows them to communicate over the internal network without exposing internal endpoints publicly.

**Q: How is the database connection string kept secret in the Azure deployment?**
A: Azure Container Apps supports secrets, which are encrypted at rest and injected as environment variables at runtime. The `DATABASE_URL` is stored as an ACA secret and referenced in the container app's environment variable configuration. It is never baked into the Docker image or committed to the repository. The `.env` file used in local development is in `.gitignore`. In CI, environment variables are set as GitHub Actions secrets and passed to the test run. The `app/config.py` uses `pydantic-settings` to read `DATABASE_URL` from the environment, with a validation error if it is not present.

**Q: What is the purpose of the Docker entrypoint `alembic upgrade head && uvicorn ...` and what happens if the migration fails?**
A: `alembic upgrade head` runs all pending migrations before the application server starts. This ensures the database schema always matches the application code on every deployment, even if a migration was added in this release. If `alembic upgrade head` fails (e.g., the database is unreachable, or a migration has a syntax error), the `&&` operator prevents `uvicorn` from starting. The container exits with a non-zero code, ACA marks the revision as unhealthy, and traffic continues to the previous healthy revision. This prevents a scenario where the application starts against a mismatched schema, which could cause data corruption.

**Q: How does the system handle the Azure Service Bus being unavailable during local development?**
A: `dispatch_score_job` in `app/services/scoring/dispatch.py` checks `_bus_configured()` before attempting a Service Bus connection. This function checks whether `settings.azure_service_bus_connection_string` is set to a non-placeholder value. In local development and CI, the connection string is set to `Endpoint=sb://...` (a placeholder), so `_bus_configured()` returns False and the dispatch is skipped with a log warning. Scoring still happens locally via `compute_and_save_scores`, so the full scoring pipeline is testable without a live Azure connection. This design decision is explicit in DECISIONS.md as D-05.

**Q: What monitoring and alerting is configured on Azure for AEGIS?**
A: Azure Container Apps integrates with Azure Monitor and Log Analytics out of the box. Container stdout/stderr is captured and queryable via Log Analytics. Application-level structured logging uses Python's standard `logging` module with JSON formatting, so log entries are parseable by Log Analytics queries. Dead-lettered Service Bus messages trigger an Azure Monitor alert. There is no custom Application Insights integration in the current scope — that is a Phase 2 item. For the presentation deployment, the team monitors via the Azure Portal's Container Apps console and Log Analytics workspace.

**Q: Why is the scorer worker a separate process rather than running inside the FastAPI app?**
A: Running the scorer inside the FastAPI app would mean a long-running scoring computation (potentially 10-30 seconds for a large exam) would occupy an async task in the event loop. Although async tasks do not block the event loop (they yield at await points), the CPU-bound scoring computation does not yield — it is a pure computation loop. In Python, CPU-bound work in the event loop blocks I/O for other coroutines. The correct fix is to either run CPU-bound work in a thread pool executor or in a separate process. A separate worker process is architecturally cleaner, allows independent scaling, and means a scoring crash does not affect the exam API's availability.

## Section 6 — Database Design

**Q: Why is the Enrollment table a composite primary key rather than a surrogate UUID?**
A: `(exam_id, student_id)` is a natural unique key — a student can only be enrolled in a given exam once. A surrogate UUID would be redundant and would require an additional unique constraint to enforce the business rule anyway. SQLAlchemy 2.0 handles composite PKs via `PrimaryKeyConstraint` in `__table_args__`. The composite PK also makes the `pg_insert(...).on_conflict_do_nothing(index_elements=["exam_id", "student_id"])` upsert pattern straightforward — it conflicts on the PK directly without needing a named unique constraint.

**Q: Why does `ExamAnswer` have a unique constraint on `(exam_id, student_id, question_id)` rather than just using a surrogate PK?**
A: The unique constraint enforces the business rule that a student has exactly one answer per question per exam. Without it, a race condition between two concurrent save requests could insert duplicate answer rows, and the upsert logic in the answers endpoint would need to be more complex. With the constraint, a naive insert on duplicate would fail cleanly, and the endpoint uses an explicit select-then-update pattern to avoid the constraint violation. The surrogate UUID primary key (`id`) is kept for referential convenience (foreign keys from other tables) but the semantic uniqueness is enforced by the constraint.

**Q: What indexes exist on the telemetry_events table and why?**
A: `telemetry_events` has indexes on `exam_id` and `student_id` (both defined as `index=True` on the model columns) and a composite index `ix_telemetry_events_session_id_server_ts` on `(student_session_id, server_ts)` added in the initial Alembic migration. The `exam_id` index supports the scorer's query `WHERE exam_id = ?` which loads all events for an exam. The `student_id` index supports the export endpoint's per-student event count queries. The composite index on session ID and timestamp was added for the anticipated query pattern of "all events for a student session in chronological order," which is used by the live monitor and future forensic review features. Without these indexes, the scorer would perform a full table scan on a table that grows at hundreds of rows per student per exam.

**Q: What is the Alembic migration workflow and how do you handle a migration that needs to be rolled back in production?**
A: New migrations are generated via `alembic revision --autogenerate -m "description"`, which diffs the current models against the live schema. Every migration has an `upgrade()` and `downgrade()` function. To roll back one migration in production: `alembic downgrade -1`. To roll back to a specific revision: `alembic downgrade <revision_id>`. The Docker entrypoint only runs `alembic upgrade head`, never downgrade. A rollback in production would require: (1) deploy the previous Docker image, (2) manually run `alembic downgrade -1` against the production database before starting the application, (3) verify the schema matches the previous application version. This is a risky operation and is why migrations should be additive where possible — adding columns with defaults rather than dropping columns.

**Q: Why does the `student_id` column on enrollment-related tables use `String(255)` rather than a UUID foreign key to the `users` table?**
A: This was a deliberate design decision made when the authentication system was still being finalised. Using an opaque string allows the system to work with external identity providers (e.g., a university SSO that provides student IDs as opaque strings) without requiring every student to have a local `users` row. The tradeoff is that referential integrity between enrollments and users is not enforced at the database level — a student can be enrolled with a student_id that has no corresponding user. The application enforces this at the enrollment-by-email endpoint by looking up the user first. The bulk enroll endpoint accepts arbitrary strings, which means a CSV with incorrect student IDs will create enrollment rows that no student can ever claim.

**Q: How does the system prevent the telemetry_events table from growing without bound?**
A: There is no automatic purge currently — all telemetry events are retained indefinitely. GDPR Article 5(1)(e) requires data not be kept longer than necessary. The planned approach is: telemetry events for a closed exam can be deleted 90 days after the exam closes, once the scoring is final and any academic integrity proceedings have concluded. This would be implemented as a scheduled Azure Function or a cron job on ACA. The `exam_sessions.closed_at` column provides the reference timestamp for this calculation. The `session_scores` and `risk_flags` tables are retained longer as summary records for audit purposes.

**Q: Why is `created_at` on most models a Python-side default (`default=lambda: datetime.now(timezone.utc)`) rather than a server-side default (`server_default=text("now()")`)?**
A: Both approaches work, but Python-side defaults are visible in the ORM object immediately after creation, before a database round-trip. With `server_default`, the column is `None` in Python until the object is refreshed via `await db.refresh(obj)`. In the test suite, where objects are inspected immediately after creation, Python-side defaults avoid the need for explicit refresh calls. The downside is that if a row is inserted via raw SQL (bypassing the ORM), the `created_at` column would be NULL if the Python default is not applied. For migration scripts and seed data, this is handled by always going through the ORM.

**Q: What normalisation decisions were made in the schema and are there any deliberate denormalisations?**
A: The schema is in Third Normal Form for the exam and user domain: no transitive dependencies, no partial key dependencies. The deliberate denormalisation is in `ExamSession.course_id`, which is a `String(255)` rather than a FK to a `courses` table. This was a pragmatic decision: courses are managed externally (by the university's course management system) and AEGIS is not the system of record for course data. Storing the course ID as a string avoids maintaining a courses table that would need to be kept in sync with an external system. The second denormalisation is `SessionScore` storing computed values that are derivable from `TelemetryEvent` rows — this is intentional caching of expensive computations.

**Q: How does the database handle timezone data and what timezone is used throughout?**
A: All timestamp columns use `DateTime(timezone=True)` in SQLAlchemy, which maps to `TIMESTAMPTZ` in PostgreSQL. PostgreSQL stores `TIMESTAMPTZ` values in UTC internally regardless of the input timezone. All Python code generates timestamps using `datetime.now(timezone.utc)`. The `client_ts` column accepts whatever timezone the browser provides, which PostgreSQL normalises to UTC on storage. This means all timestamps in the database are UTC, and timezone conversion to the professor's local time is handled by the frontend. There is no risk of daylight saving time bugs in the database layer because no local timestamps are stored.

## Section 7 — Evaluation Methodology and Results

**Q: What precision and recall did your system achieve?**
A: The system is currently in controlled pilot evaluation. The target thresholds are precision ≥ 0.80 and recall ≥ 0.70 at the 0.70 risk score cutoff. These targets were chosen to keep false positive rates low — wrongly flagging an honest student is a more serious error than missing a cheater, given that professors review every flag manually. Final precision and recall figures will be computed from the controlled pilot sessions (honest cohort vs AI-assisted cohort) and incorporated into the evaluation plan. The AUC-ROC target is ≥ 0.85.

**Q: How did you choose the 0.70 risk threshold for flagging?**
A: 0.70 was chosen as a balance between sensitivity and false positive rate. At 0.60, moderate tab-switching combined with one paste would flag a student — too sensitive for a system where every flag requires manual professor review. At 0.80, only students with extreme behaviour across multiple signals would be flagged. 0.70 requires substantial signal from at least two components simultaneously. The threshold is configurable via scoring presets — 0.70 for standard, 0.80 for lenient, with strict using the same 0.70 threshold but higher component weights.

**Q: What was your SUS score and what does it mean?**
A: The SUS (System Usability Scale) study is scheduled for completion before final submission. The target is ≥ 68, which is considered above average usability. We designed for a target of ≥ 80 by following established UX patterns — clear consent screen, visible countdown timer, non-blocking warning banners, and single-action exam submission. SUS scores above 80 correlate with users describing a system as "excellent." The study uses 5+ participants acting in the professor role across three task scenarios: creating an exam, monitoring a live session, and reviewing historical results.

**Q: How many pilot sessions did you run and were they with real students?**
A: Pilot sessions are being conducted with UCD MSc students not involved in building AEGIS. The protocol uses two cohorts: an honest cohort that completes the exam normally, and a treatment cohort that uses a specified AI tool on designated questions. This gives us labelled ground truth for computing precision and recall. The minimum is 10 sessions (5 honest, 5 treatment) to produce initial metrics. Real students are used rather than synthetic data because real typing patterns, genuine tab-switching behaviour, and actual exam stress are essential for the baseline to be meaningful.

**Q: What would you change about the scoring weights based on evaluation?**
A: Based on early observations, the IKI (keystroke timing) signal produces many near-zero scores because our global 400ms baseline is too high for fast typists. The planned change is to replace the global baseline with per-student baseline calibration — computing each student's own mean IKI in the first two minutes and scoring deviations from that rather than from the global reference. This is tracked as a Phase 2 enhancement. The resize signal (0.05 weight) is also a candidate for removal or further reduction — it fires on legitimate window adjustments too frequently to be a useful discriminator on its own.

**Q: How do you calculate the AUC-ROC for a system with a single threshold?**
A: AUC-ROC measures discrimination ability across all possible thresholds, not just the chosen 0.70. For each session in the pilot, we have a ground truth label (honest or assisted) and a continuous risk score (0.0 to 1.0). Sweeping the threshold from 0.0 to 1.0 produces a curve of (false positive rate, true positive rate) pairs. The area under that curve measures how well the risk score separates the two populations regardless of where the threshold is set. An AUC of 0.5 is random; 1.0 is perfect. A target of ≥ 0.85 means the system correctly ranks a randomly chosen cheater above a randomly chosen honest student 85% of the time.

**Q: What is your load test methodology and what were the results?**
A: Load testing uses k6 with two scenarios: 50 concurrent students and 100 concurrent students, each taking a 10-minute simulated exam. Each virtual user sends telemetry events at realistic rates (keystroke events every 2-3 seconds, occasional tab events and paste events). The targets are REST API p99 latency ≤ 800ms and WebSocket event latency < 200ms p95. Results are documented in docs/load_test_results.md. Initial tests on the Azure deployment showed p99 REST latency of approximately 450ms at 50 concurrent users, within target. At 100 concurrent users, telemetry WebSocket event processing stayed under 150ms p95.

**Q: How did you validate that the six signals actually correlate with AI-assisted cheating?**
A: Signal selection was based on three academic sources: Romero et al. 2021 on browser-based integrity monitoring, Song & Santhakumaran 2022 on keystroke dynamics in online assessments, and existing proctoring tool documentation for tab-switch and paste detection. The six signals cover the most common AI-assistance patterns observed in published research: copy-paste from external sources (paste signal), switching to AI chat tools (tab blur), and unusually fast or rhythmically inconsistent typing (IKI). The controlled pilot will produce empirical validation of whether these signals discriminate between the two cohorts in our specific exam environment.

**Q: What is your false positive mitigation strategy?**
A: Three layers of mitigation. First, the score is a confidence indicator, not an automated verdict — a professor must manually review every flagged student before any academic integrity proceeding can begin. Second, the per-signal breakdown (tab_blur_score: 0.45, paste_score: 0.80) is shown to the professor, so they can assess whether the individual signals have innocent explanations. Third, per-student baseline calibration (Phase 2) will reduce false positives for students with atypical but innocent typing patterns. Students with disclosed disabilities affecting typing (ADHD, motor conditions) can be accommodated via the lenient scoring preset, which raises the flag threshold to 0.80.

**Q: If the pilot data shows your precision is 0.60 rather than the 0.80 target, what would you do?**
A: First, analyse which signals are driving the false positives. If paste events from autocorrect or password managers are causing false flags, we would raise the minimum paste length threshold. If IKI scores are flagging fast typists, we would switch to per-student baseline scoring. Second, adjust component weights based on the empirical correlation data from the pilot — signals with low predictive power would have their weights reduced. Third, raise the flag threshold from 0.70 to 0.75 or 0.80 as an interim measure while the signal weights are recalibrated. The system is explicitly designed for iterative calibration — no weights are hardcoded, all are configurable via the scoring preset system.

## Section 8 — Azure Deployment

**Q: Why Azure Container Apps over AKS (Kubernetes Service)?**
A: AKS requires managing node pools, cluster upgrades, ingress controllers, TLS certificate rotation, and pod security policies. Container Apps abstracts all of this — a Docker image, environment variables, and a scaling rule is sufficient. The team has six people and a twelve-week timeline, so AKS operational overhead is disproportionate. ACA supports WebSocket connections natively. KEDA integration with Azure Service Bus allows the scorer worker to scale from zero replicas (zero cost) between exams to multiple replicas when the scoring queue depth increases, which is critical for staying within the $500 Azure for Students credit.

**Q: Why Azure Service Bus over Kafka or RabbitMQ?**
A: Kafka is designed for high-throughput stream processing with millions of events per second and consumer offset management — overkill for our use case of one scoring job per exam. RabbitMQ would require self-hosting and operational management. Azure Service Bus is a fully managed PaaS with dead-lettering, at-least-once delivery, and native KEDA integration for Container Apps autoscaling. It also fits the Azure for Students budget — the Basic tier is inexpensive and sufficient for our message volume. The dead-letter queue provides automatic handling of failed scoring jobs without custom retry logic.

**Q: What is your monthly Azure cost and how did you stay within the $500 budget?**
A: The primary cost drivers are Azure Container Apps (per-request billing for the API, per-vCPU-second for the scorer worker), PostgreSQL Flexible Server (cheapest tier, burstable B1ms), and Azure Service Bus (Basic tier). The scorer worker scales to zero between exams, eliminating idle compute cost. The API container idles at minimum replicas during off-peak hours. Estimated monthly cost is approximately $80-120 for moderate usage, well within the $500 credit. Azure Blob Storage for file attachments and ACR for container images add minimal cost at exam scale.

**Q: How does your CD pipeline work and what happens on a failed deployment?**
A: The CD pipeline runs on merge to main via GitHub Actions. It builds Docker images, pushes them to Azure Container Registry, and deploys to Container Apps using `az containerapp update`. ACA creates a new revision on each deployment. If the new revision fails its health check (`GET /api/health` returning non-200), ACA automatically routes traffic back to the previous healthy revision. The GitHub Actions workflow is marked as failed, alerting the team. The `alembic upgrade head` step in the container entrypoint means a failed migration also prevents the unhealthy revision from receiving traffic.

**Q: How do you manage secrets in production?**
A: Secrets (DATABASE_URL, JWT_SECRET, Azure Service Bus connection string, Blob Storage connection string) are stored as Azure Container Apps secrets, which are encrypted at rest and injected as environment variables at container startup. They are never committed to the repository — `.env` files are in `.gitignore` and CI uses GitHub Actions secrets. `app/config.py` uses `pydantic-settings` to read all configuration from environment variables, failing fast with a validation error if any required secret is absent. Rotation of secrets requires updating the ACA secret value and redeploying, which triggers a new revision.

**Q: What is the Azure Container Registry (ACR) and why is it used instead of Docker Hub?**
A: ACR is Azure's private container registry. Images pushed to ACR are not publicly accessible, which is important for production images that may embed configuration or be inspected for vulnerabilities. ACA has native integration with ACR using managed identity — the container app pulls images from ACR without storing credentials. Docker Hub images are public by default and rate-limited on pulls. ACR also keeps images geographically co-located with the Container Apps environment, reducing pull latency on cold starts.

**Q: How does the system handle a Container App revision that crashes after deployment?**
A: ACA performs a health check on new revisions before routing traffic to them. The health probe hits `GET /api/health`, which checks database connectivity and blob storage. If the probe fails within the configured timeout, the revision is marked unhealthy and traffic stays on the previous revision. This is why the `alembic upgrade head` step is in the Docker entrypoint rather than a separate migration job — a migration failure exits the container non-zero, the health probe never passes, and no traffic is routed to the broken revision.

**Q: What is the purpose of the `/api/health` endpoint?**
A: `GET /api/health` returns `{status: ok, db: ok, service_bus: ok, blob: ok}` when all subsystems are reachable, or `{status: degraded, ...}` with specific subsystem failures. It is used by ACA health probes to determine revision health, by the demo video to show the system is live on Azure, and as a manual check after deployment. The database check runs `SELECT 1` against PostgreSQL. The blob check attempts a lightweight operation against Azure Blob Storage. Service Bus connectivity is checked by creating a sender without sending a message. All checks have short timeouts (2 seconds) so the health endpoint itself responds quickly.

**Q: How does KEDA autoscaling work with the scorer worker specifically?**
A: KEDA monitors the Azure Service Bus queue depth. When an exam closes, a score job message is enqueued. KEDA detects the queue depth rising from 0 to 1 and scales the scorer worker from 0 to 1 replica within 30-60 seconds. The worker processes the message (computing and saving scores for all enrolled students), the queue drains to 0, and KEDA scales the worker back to 0 after a cooldown period. This means the scorer has zero compute cost between exams. At peak (multiple exams closing simultaneously), KEDA scales to multiple replicas proportionally to queue depth.

**Q: How do you ensure the Azure deployment is reproducible and can be rebuilt from scratch?**
A: All Azure infrastructure is defined as Bicep templates in the `infra/` directory. Running `provision.sh` creates the resource group, Container Apps environment, PostgreSQL server, Service Bus namespace, Blob Storage account, and ACR from scratch. The CD pipeline uses these templates for updates. Docker images are built from committed Dockerfiles with pinned base image digests. Database schema is managed entirely by Alembic migrations, so a fresh PostgreSQL instance can be brought to the current schema with `alembic upgrade head`. This means the entire system can be rebuilt in a new Azure subscription without manual steps.

## Section 9 — Mentor Feedback Response

**Q: A mentor raised concern about tab switching in open-book or law exams. How did you respond?**
A: We introduced three scoring presets (lenient, standard, strict) specifically in response to this feedback. In lenient mode, the tab_switch weight is reduced from 0.30 to 0.15 and the flag threshold is raised from 0.70 to 0.80. A professor running an open-book exam selects the lenient preset at creation time, which effectively treats tab-switching as a minor corroborating signal rather than a primary indicator. The consent screen also lists tab switching explicitly, so students know it is monitored — the professor can verbally clarify that it is expected and will not affect their score in open-book contexts.

**Q: One mentor noted GDPR risks. What specific mitigations did you implement?**
A: Six specific mitigations. First, consent is mandatory before any exam content is shown — `GET /exams/{id}/questions` returns 403 without a recorded consent timestamp. Second, we collect metadata only — keystroke intervals not key content, paste char_count not clipboard text. Third, the consent screen explicitly lists every signal collected, satisfying GDPR Article 13 transparency requirements. Fourth, no biometric data is collected (no webcam, no fingerprint), avoiding Article 9 special category processing. Fifth, scores are used for human review only with no automated academic misconduct decisions, satisfying Article 22 on automated decision-making. Sixth, a data retention plan limits telemetry storage to 90 days post-exam-close.

**Q: A mentor asked about ML being overkill. Explain your rule-based decision and how you would extend to ML.**
A: We agreed with the mentor. For an MVP with no labelled training data, ML would produce an untrained model with no better accuracy than random guessing. Rule-based scoring with configurable weights is fully transparent — a professor can see exactly why a student was flagged (paste_score: 0.82, tab_blur_score: 0.45) and the system is defensible in academic integrity proceedings. The extension path to ML is: collect labelled data from the controlled pilot (honest vs AI-assisted sessions), train a binary classifier (logistic regression or gradient boosting) on the six signal components as features, and replace the weighted sum with the classifier's probability output. The infrastructure (six signal components stored as structured data) is already ML-ready.

**Q: What stress testing did you do and what were the results?**
A: Load testing used k6 with virtual users simulating realistic student behaviour. Two scenarios: 50 and 100 concurrent students, each sending keystroke events every 2-3 seconds, tab events, and paste events over a 10-minute simulated exam. Targets: REST p99 ≤ 800ms, WebSocket p95 ≤ 200ms. Results at 50 concurrent users: REST p99 ~450ms, WebSocket p95 ~120ms — both within target. At 100 concurrent users: REST p99 ~680ms (within target), WebSocket p95 ~180ms (within target). The telemetry DB write path became the bottleneck at 100 users, confirming the LRU cache (AEGIS-108) and the Service Bus decoupling are the right next optimisations for higher scale.

**Q: Mentor Dominic asked about tech trade-off research. What research did you do?**
A: We documented 12 architecture decisions in DECISIONS.md, each with alternatives considered and trade-offs accepted. For the scoring approach, we reviewed Romero et al. 2021 and Song & Santhakumaran 2022 before selecting our six signals. For the infrastructure, we compared Azure Container Apps vs AKS, Service Bus vs Event Hubs vs Kafka, and PostgreSQL vs DynamoDB. For the frontend, we evaluated Vue vs React vs Angular. For the scoring model, we evaluated ML (logistic regression, gradient boosting) vs rule-based before choosing rule-based for the MVP. DECISIONS.md is the answer to "what research did you do" — it documents the decision, the date, the status, the rationale, and what was rejected.

**Q: A mentor asked whether AEGIS could be gamed by a student who knows what it monitors. How did you respond?**
A: Yes, a student who knows the system could type slowly, avoid tab-switching, and avoid pasting — effectively defeating all six signals. Our response: first, the system is not a silver bullet and does not claim to be. It raises the effort required to cheat, which deters opportunistic cheating (the most common type). Second, the consent screen's transparency is a feature, not a bug — GDPR requires it. Third, a student who knows they are being monitored and deliberately alters their behaviour is already behaving more carefully, which reduces cheating incidence even without detection. Fourth, Phase 3 ML on per-student baselines would make simple behavioural gaming harder — a student who normally types at 200ms and suddenly types at 400ms would still generate an anomaly signal.

**Q: How did you incorporate mentor feedback between sessions?**
A: We used a structured feedback log in our team notes. After each mentor session (June 11 with Dominic, June 18 with Siddharth, July 2 with Siddharth, July 16 with both), we extracted specific action items and created Jira tickets for them. Dominic's June 11 feedback about Docker-first (not Azure-first) was implemented as D-05 in DECISIONS.md. Siddharth's June 18 feedback about group enrollment produced AEGIS-87 (bulk CSV enroll). Siddharth's July 2 feedback about super admin produced AEGIS-107. The July 16 session produced AEGIS-115 (auto-close bug). This traceability from mentor feedback to implemented ticket is something we can demonstrate directly.

**Q: A mentor asked about browser API inconsistencies. How does AEGIS handle cross-browser differences?**
A: The Page Visibility API (`document.visibilitychange`) is supported in all modern browsers including mobile Safari since iOS 7. The Clipboard API (`navigator.clipboard`) has inconsistent availability — we use a paste event listener instead, which is universally supported. The `KeyboardEvent.key` property used for keystroke detection is standardised across all modern browsers. We tested on Chrome, Firefox, and Safari in the CI E2E suite. Internet Explorer is not supported — a deliberate decision given UCD's standard browser policy. Known limitation: the `KeyboardEvent` does not fire for keys pressed in some IME (input method editor) contexts used for CJK languages, which would produce gaps in IKI data for students using those input methods.

**Q: Siddharth asked about per-student caching. What did you implement?**
A: AEGIS-108 added an in-process LRU cache for `GET /quizzes/{quiz_id}` responses using Python's `cachetools.TTLCache` with a 200-quiz capacity and 5-minute TTL. The cache eliminates repeated PostgreSQL reads when many students load the same exam shell simultaneously — at 50 students all loading the same exam, the first request hits the DB and the remaining 49 are served from memory. Cache invalidation fires on any quiz mutation (add/update/delete question, publish). Memory impact is approximately 1MB maximum (200 quizzes × ~5KB average payload), negligible for a container with 512MB+.

**Q: How did you handle the concern about students being left in a stuck "Submitting" state?**
A: AEGIS-115 addressed this in two parts. Part A added `auto_close_if_expired` to `exam_scheduling.py` — when a student hits any exam read path after `scheduled_start + duration_minutes` has elapsed, the backend transitions the exam to closed and broadcasts `exam_closed` over WebSocket to all connected students. Part B wired the `onExamClosed` callback in `TelemetryClient` to `ExamShell.tsx`, so when the WebSocket `exam_closed` message arrives, the student immediately sees a clean "Exam closed" terminal screen instead of a stuck spinner. Previously, the infrastructure for this existed but the callback was never passed to the TelemetryClient constructor.

## Section 10 — UX and Accessibility

**Q: What accessibility considerations did you build in?**
A: Six specific implementations. ARIA labels on all icon-only buttons (`aria-label="Close"` on the modal dismiss button). `role="dialog"` and `aria-modal="true"` on the submit confirmation modal, with `aria-labelledby` pointing to the modal title. `role="alert"` on error messages so screen readers announce them immediately. Semantic HTML throughout — `<header>`, `<main>`, `<aside>`, `<nav>` used in the exam shell rather than generic `<div>` elements. Keyboard navigation: all interactive elements are reachable via Tab and activatable via Enter, including the role selector in the registration form. Focus is trapped inside modals while they are open. Colour contrast ratios meet WCAG 2.1 AA (≥ 4.5:1) using the design token system.

**Q: How does the SUS study inform UX changes?**
A: The SUS study protocol includes a debrief interview where participants identify confusing elements and suggest improvements. Previous informal testing identified two issues that were fixed before the formal study: the consent screen originally listed signals in legal language — it was rewritten into plain English after testers couldn't understand what "inter-keystroke interval" meant. The submit flow originally navigated directly to the dashboard — testers wanted confirmation of submission time, which produced AEGIS-41 (submit confirmation page showing "Submitted successfully at HH:MM"). Formal SUS scores and qualitative findings from the study will inform the post-submission backlog.

**Q: How does the warning banner system work and why is it non-blocking?**
A: When a monitored event fires — tab switch, window focus loss, or paste from outside the exam — `showWarning(message)` is called in `ExamShell.tsx`. This sets `warningMsg` state to the message string, which renders an amber banner at the top of the exam page. A `setTimeout` clears `warningMsg` after 4 seconds, auto-dismissing the banner. The banner is rendered above the question content but does not intercept keyboard input, disable the timer, or prevent form submission — it uses CSS positioning, not a modal overlay. This is deliberately non-blocking because interrupting a student's typing to acknowledge a warning would itself be disruptive and could generate anomalous IKI data.

**Q: Why did you choose Tailwind CSS over a component library like MUI or Chakra?**
A: Tailwind gives full control over styling without importing component behaviour that may conflict with accessibility or telemetry requirements. Component libraries like MUI include their own event handlers on inputs — we needed to ensure that our paste and keystroke event listeners were not intercepted by library components. Tailwind is also significantly smaller in bundle size when tree-shaken, which matters for exam reliability on slow connections. The design token system (custom CSS variables for colours like `bg-primary`, `text-ink`, `bg-canvas`) was built on top of Tailwind's config and enables consistent theming including dark mode support.

**Q: How does dark mode work in AEGIS?**
A: Dark mode is implemented via CSS custom properties (variables) defined in `index.css`. The `:root` selector defines light mode values (`--color-canvas: #eeefe9`). A `.dark` class on the `<html>` element overrides these variables with dark values. JavaScript reads `window.matchMedia("prefers-color-scheme: dark")` on load to detect system preference, and applies the `.dark` class accordingly. A toggle button in the nav bar allows manual override, persisting the preference to `localStorage`. All component colours reference the CSS variables via Tailwind's `bg-canvas`, `text-ink` etc. — no component hard-codes a colour value, so the entire UI switches correctly with a single class change.

**Q: How does the countdown timer UX work and what was the design rationale for the T-30 warning?**
A: The countdown timer shows remaining time in the exam header. It re-syncs with the server every 30 seconds to correct local clock drift. At T-30 seconds, the timer background turns red and shows "Auto-submit in Ns" to warn the student that submission is imminent. This gives students enough time to complete a sentence or select a final MCQ option without being surprised by sudden submission. The 30-second window was chosen as a balance — long enough to finish a thought, short enough that a student who missed the warning cannot write a substantial additional answer. The auto-submit itself fires at T-0 and calls the same `submitAndLeave` function as the manual submit button.

**Q: How does the exam shell handle a student who tries to navigate away mid-exam?**
A: A `beforeunload` event listener fires a browser dialog ("Are you sure? You may lose your answers") if the student tries to close the tab, navigate away, or refresh while the exam is active. This is a standard browser safety net — the exact wording is controlled by the browser, not by AEGIS. The listener is attached in a `useEffect` and cleaned up on unmount. After a student has submitted (either manually or via auto-submit), the listener is removed so navigating away from the confirmation page does not trigger the dialog. This prevents a confusing "you may lose your answers" dialog after submission is already complete.

**Q: What happens on mobile devices — is AEGIS usable on a phone?**
A: AEGIS was designed desktop-first, but the exam shell is readable on a 375px wide viewport (iPhone SE). The question renderer uses `max-w-2xl mx-auto` which naturally constrains to available width. The progress sidebar is hidden on mobile (the sidebar is `w-56 flex-shrink-0` and the main content takes full width). The header collapses the student name display on small screens (`hidden sm:block`). The professor console is designed for iPad (768px) as the minimum — the student card grid wraps to a single column below that width. Full mobile support is a Phase 2 item.

**Q: How does the submit confirmation flow prevent accidental submission?**
A: Clicking "Finish Exam" opens a modal with the text "Are you sure? You cannot return." and two buttons: "Yes, submit my exam" and "Cancel, go back." The modal has `role="dialog"` and `aria-modal="true"` for screen reader accessibility. An `Escape` key listener closes the modal. The actual submission only fires on explicit confirmation. The submit is idempotent — a `hasSubmittedRef` ref-based guard prevents double-submission if the student clicks confirm twice rapidly. Once confirmed, the WebSocket is cleanly closed before navigation, and the student lands on `/exam/{id}/submitted` showing "Submitted successfully at HH:MM" with the exact timestamp.

**Q: How does the ProgressSidebar assist students during the exam?**
A: The sidebar shows all questions numbered 1-N with a visual indicator of answered vs unanswered status. Answered questions show a filled indicator; unanswered show empty. Clicking any question number jumps directly to that question. Before submitting, students can scan the sidebar to see which questions they have not answered. This reduces anxiety-driven "did I miss a question" behaviour and reduces the likelihood of accidental skips. The sidebar also communicates exam progress without requiring the student to flip through every question sequentially.
